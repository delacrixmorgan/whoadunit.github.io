# Data Problem

This document describes the current state of the data pipeline and the mismatches found between source files, scripts, and the JSON files the app consumes.

---

## Summary

The data pipeline is broken in multiple ways. The scripts, the base spreadsheet, the documentation, and the live JSON files are all out of sync with each other. The per-year JSON files the app actually reads (`representatives_2022.json`, `representatives_2023.json`) were generated by an old process and no longer match the source XLSX files.

---

## 1. What the app actually reads

The web app (`useRepresentatives.js`) loads data from:

```
public/data/representatives-manifest.json   → [2023, 2022]
public/data/representatives_2023.json       → flat array of ADUN records
public/data/representatives_2022.json       → flat array of MP records
```

Each record in a year file is a **flat object** with fields:
`type`, `federalSeatCode`, `federalSeatName`, `stateSeatCode`, `stateSeatName`, `state`, `electedYear`, `name`, `party`, `gender`, `address`, `email[]`, `phoneNumber[]`, `facebook[]`, `twitter[]`, `pollingDistricts[]`

---

## 2. What the scripts produce (and what's broken)

### Script 1: `mw-to-xlsx.cjs`

- **Input:** `data/mw/results_YEAR.mw`
- **Output:** `data/xlsx/representative_YEAR.xlsx` (flat, single-sheet, one row per representative)
- **Status:** Works as designed. Produces the year xlsx files.

### Script 2: `convert-xlsx-to-json.cjs`

- **Input:** `data/xlsx/representative_base.xlsx` — expects **two sheets: "MP" and "ADUN"**
- **Output:** `public/data/representatives.json` — nested format (`{ mp: {...}, aduns: [...] }`)
- **Status:** ❌ Broken. `representative_base.xlsx` no longer has "MP" or "ADUN" sheets. It has one sheet called **"Representatives"** with 611 rows where the `type` column is **entirely empty**. The convert script would produce no output if run.

### Script 3: `split-by-year.js`

- **Input:** `public/data/representatives.json` (nested format)
- **Output:** `public/data/representatives_YYYY.json` (flat per-year files)
- **Status:** ❌ Broken. `public/data/representatives.json` **does not exist** — the convert script has never successfully run against the current base.xlsx. Also, `package.json` references this script at `scripts/split-by-year.js` but the file is actually at `public/data/scripts/split-by-year.js`.

### No script for the most important step

There is **no script** that converts `representative_2022.xlsx` / `representative_2023.xlsx` directly into `representatives_2022.json` / `representatives_2023.json`. This is the step the app actually needs.

---

## 3. `representative_base.xlsx` state

| What it should be (per README) | What it actually is |
|---|---|
| Two sheets: "MP" and "ADUN" | One sheet: "Representatives" |
| `type` column blank (implied by sheet) | `type` column present but **all values empty** |
| 222 MP rows + ~450 ADUN rows | 611 rows, all with `type = null` |

The base spreadsheet has been restructured but not connected to any working script.

---

## 4. Stale per-year JSON files

The `representatives_2022.json` and `representatives_2023.json` files in `public/data/` were generated by an **old process** that no longer exists. They do not match the current XLSX source files.

### Concrete example — Lim Guan Eng (MP, 2022):

| Field | `representative_2022.xlsx` | `representatives_2022.json` |
|---|---|---|
| `federalSeatCode` | `P.043` (normalised: `P043`) | `P043` ✓ |
| `stateSeatCode` | **`null`** (no state seat for MPs) | **`N07`** ← wrong |
| `stateSeatName` | **`null`** | **`Sungai Puyu`** ← wrong |
| `pollingDistricts` | 24 districts (full federal constituency) | 8 districts (one state seat only) ← wrong |
| `state` | `Pulau Pinang` | `Penang` (normalised) |

The JSON has `stateSeatCode: N07` and the short polling-district list because it was generated from an old source that included the EC's representative state seat for each MP row. The current XLSX correctly leaves those blank for MPs.

---

## 5. Documentation is outdated

Both `data/scripts/README.md` and `data/md/DATA_SCHEMA.md` describe the old nested JSON format (`representatives.json` with `mp` and `aduns` nested inside each seat object). The app no longer uses that format.

---

## 6. App-level bugs caused by the stale data

Because the JSON has `stateSeatCode: N07` on MP records, the `ProfilePage` incorrectly renders a "State Seat" row on MP profile pages. This is a code bug (the page should guard on `rep.type === 'ADUN'`), but it is made worse by the stale data.

The more serious bug is in `DirectoryPage.jsx`: ADUN profile links are constructed using only `stateSeatCode` (e.g. `/profile/2023/N23`), not the globally unique `federalSeatCode + stateSeatCode`. Since state seat codes restart per state, this causes the wrong representative to be shown — for example, searching Lim Guan Eng (ADUN N23 Air Putih under P048) navigates to Ahmad Sulaiman (ADUN N23 Belantek under P013) because P013 sorts before P048.

---

## What needs to be fixed

1. **Write a new conversion script** that reads `representative_YEAR.xlsx` directly and outputs `representatives_YEAR.json` in the flat format the app expects — including normalising codes (strip dot), gender (`Male` → `M`), and splitting semicolon-delimited fields into arrays.

2. **Fix `representative_base.xlsx`** — either restore the MP/ADUN two-sheet structure or update the convert script to match the new single-sheet layout and populate the `type` column.

3. **Fix `DirectoryPage.jsx`** — ADUN profile links must use the 3-param URL `/profile/:year/:federalSeatCode/:stateSeatCode`, not 2-param.

4. **Fix `ProfilePage.jsx`** — the "State Seat" row in Seat Details should only render when `rep.type === 'ADUN'`.

5. **Update `data/scripts/README.md`** and **`data/md/DATA_SCHEMA.md`** to reflect the actual pipeline and data format.
